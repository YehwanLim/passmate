# 리포트 요약과 실무자 코멘트 가독성 설계

## 목표

분석 결과의 첫 화면에서 지원자의 정체성이 한눈에 읽히게 하고, 긴 실무자 코멘트를 스캔 가능한 에디토리얼 블록으로 나눠 보여준다.

## 범위

기존 JSON 스키마는 변경하지 않는다. `firstImpression.persona`, `firstImpression.summaryOneLiner`, `pmComment`는 모두 기존 문자열 필드를 유지한다.

## 생성 규칙

- `persona`는 "도메인 + 행동 + 특징" 구조를 유지하되 공백 포함 28자 이내, 한 문장으로 작성한다.
- `summaryOneLiner`는 지원자 소개를 반복하지 않고 채용 담당자 관점의 핵심 인상을 공백 포함 42자 이내, 한 문장으로 작성한다.
- 둘 다 나열식 접속, 두 개 이상의 경험 설명, 지원 의지 문장으로 길어지는 구성을 금지한다.
- `pmComment`는 정확히 세 문단으로 작성한다. 문단은 빈 줄(`\n\n`)로 구분하며, 순서대로 지원서에서 읽힌 관통 인상, 더 선명하게 만들 연결고리, 면접에서 준비할 구체적 설명을 다룬다.
- 각 `pmComment` 문단은 1~2문장으로 제한하고, 같은 사례나 평가를 반복하지 않는다.

## 화면 표현

- 상단 제목은 기존 `persona`를 사용하며, 긴 구형 데이터도 화면에서 안전하게 줄 수가 제한되도록 처리한다.
- 상단 보조 문장은 기존 `summaryOneLiner`를 사용하며, 긴 구형 데이터는 읽는 흐름을 해치지 않는 선에서 줄 수가 제한되도록 처리한다.
- 실무자 코멘트의 빈 줄 기준 세 문단을 `읽힌 인상`, `더 선명해질 지점`, `면접에서 준비할 것` 블록으로 표시한다.
- 구형 데이터처럼 문단 구분이 없는 코멘트는 문장 경계로 최대 세 조각을 만들어 기존 리포트도 읽기 좋게 표시한다.
- 각 블록의 제목과 핵심 문장은 대비 있는 텍스트 색과 굵기로 표시한다. 본문 전체를 과도하게 강조하거나 형광색 효과를 쓰지 않는다.

## 검증

- 프롬프트 계약 테스트는 글자 수 제한과 세 문단 규칙을 확인한다.
- 코멘트 분할 도우미는 문단 입력과 단일 긴 문단 입력을 모두 세 블록으로 정리하는지 테스트한다.
- 리포트 화면 테스트는 세 제목과 강조 가능한 블록 구조를 확인한다.
