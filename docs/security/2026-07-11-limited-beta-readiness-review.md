# 제한 베타 출시 전 보안·기능 검토 보고서

검토일: 2026-07-11

대상: PassMate (`job_prep_platform`)

판정: **출시 보류 (NO-GO)**

## 1. 결론

현재 상태로는 실제 사용자 자소서를 처리하는 제한 베타를 시작하면 안 된다. 출시 차단 이슈 5건이 확인됐다.

1. 사용자 데이터 API에 인증과 객체 소유권 검사가 없어 다른 사용자의 프로젝트와 자소서 원문을 조회할 수 있다.
2. 유료 Gemini 호출 API가 인증·속도·비용 제한 없이 공개돼 비용 남용이 가능하다.
3. 프로덕션에 포함되는 DB 테스트 API를 GET으로 호출하면 테스트 데이터가 계속 생성된다.
4. 피드백 API가 요청자가 보낸 `userId`를 신뢰해 임의 사용자 생성과 타 분석 피드백 조작을 허용한다.
5. Supabase RLS 활성화 및 정책이 저장소에 적용 가능한 형태로 존재하지 않아 관리자 화면의 직접 DB 접근을 안전하다고 입증할 수 없다.

추가로 타입 검사 실패, 분석 결과 미영구 저장, API 실패 시 목 데이터 표시, 삭제 실패 후 화면에서만 삭제, 개인정보 처리·삭제 정책 부재가 확인됐다.

## 2. 검토 범위와 방법

- Vercel 라우팅, `api/` 서버리스 함수, Express 서버, Supabase 클라이언트와 관리자 데이터 훅을 정적으로 추적했다.
- Prisma 스키마, SQL 스키마와 역할 마이그레이션을 비교했다.
- 인증, 객체 소유권, 입력 경계, CORS, 유료 API 남용, 개인정보 저장, 테스트 엔드포인트와 오류 처리를 검토했다.
- Git 전체 이력에서 일반적인 Gemini·Supabase service role·DB URL 비밀 패턴을 검색했다. 일치하는 추적 파일은 발견되지 않았다. 이는 별도 비밀 스캐너의 대체가 아니다.
- `npm run check`, `npm run build`, `npm audit --json`, `npm audit --omit=dev --json`을 실행했다.

### 제한사항

배포 URL과 분리된 일반 사용자 A/B·관리자 테스트 계정이 제공되지 않아 운영 Supabase 정책과 실제 배포본에 대한 동적 침투 테스트는 수행하지 않았다. 아래 재현은 코드상 확정 가능한 요청 흐름이며, 수정 후 스테이징 환경에서 A/B 계정으로 다시 검증해야 한다. 실제 RLS 설정은 저장소의 주석 상태만으로 판단했으며 Supabase 원격 정책은 확인하지 않았다.

## 4. 발견 사항

### BLOCK-01 — 미인증 사용자 데이터 조회 및 IDOR/BOLA

- 심각도: 치명적
- 영향: 사용자 이메일과 연결된 프로젝트, 지원 기업·직무, 자기소개서 질문·원문, AI 분석 결과
- 공격 전제: 인터넷 접근만 필요하다. 프로젝트 또는 분석 UUID를 알면 상세 조회가 가능하고, `/api/projects`는 UUID 없이도 첫 번째 사용자의 목록을 반환한다.
- 증거:
  - `api/projects.js:33-57`은 인증 없이 `findFirst()`로 첫 사용자를 선택한다.
  - `api/projects/[projectId]/index.js:8-20`은 `projectId`만으로 조회한다.
  - `api/projects/[projectId]/analyses.js:8-26`은 `projectId`만으로 분석 목록을 조회한다.
  - `api/analysis/[id].js:32-58`은 `id`만으로 `inputText`와 `aiResponseJson`을 반환한다.
- 재현:
  1. 인증 헤더 없이 `GET /api/projects`를 요청한다.
  2. 반환된 프로젝트 ID로 `GET /api/projects/{id}/analyses`를 요청한다.
  3. 반환된 분석 ID로 `GET /api/analysis/{id}`를 요청한다.
  4. 응답에 `input_text`와 `ai_response_json`이 포함되는지 확인한다.
- 권장 수정: 공통 서버 인증 모듈로 Supabase 액세스 토큰을 검증하고, 모든 Prisma 쿼리에 `userId = token.sub` 소유 조건을 포함한다. 단건 조회는 복합 조건 `findFirst({ where: { id, userId } })`로 처리하고 타 사용자 객체는 404로 응답한다.
- 회귀 테스트: 미인증=401, 사용자 A의 객체를 B가 요청=404, A의 정상 요청=200을 모든 읽기·수정·삭제 API에 공통 적용한다.

### BLOCK-02 — 인증·속도·비용 제한 없는 Gemini 프록시

- 심각도: 치명적
- 영향: Gemini 비용 소진, 서비스 거부, 임의 프롬프트 처리
- 공격 전제: 인터넷 접근만 필요하다.
- 증거:
  - `api/analyze.js:402-443`은 공개 CORS를 허용하고 인증 없이 최대 6,000자의 분석을 수행한다.
  - `api/gemini.js:1-28`은 인증 없이 임의 `prompt`를 Gemini로 전달한다.
  - `api/test-gemini.js:3-18`은 GET 요청마다 Gemini를 호출한다.
  - 코드에서 사용자/IP 속도 제한, 일일 비용 한도, 중복 요청 키 또는 비상 중단 스위치를 찾지 못했다.
- 재현: 인증 없이 각 엔드포인트를 반복 호출하고, 401 또는 429 없이 외부 API 호출이 계속 발생하는지 관측한다.
- 권장 수정: `/api/gemini`와 `/api/test-gemini`를 프로덕션에서 제거한다. `/api/analyze`는 인증된 베타 사용자만 허용하고 사용자·IP별 제한, 일일 사용자 한도, 전체 비용 상한, 동시 요청 제한, 요청 ID 기반 중복 방지를 추가한다. 운영자가 즉시 분석 기능을 끌 수 있는 서버 측 환경 플래그를 둔다.
- 회귀 테스트: 미인증=401, 허용량 초과=429, 동일 idempotency key 중복=기존 결과 또는 409, 전역 중단=503을 검증한다.

### BLOCK-03 — 공개 GET 요청이 DB를 변경하는 테스트 API

- 심각도: 높음
- 영향: 테스트 프로젝트·분석·토큰 사용량 무제한 생성, DB 오염과 저장공간 소진
- 증거: `api/test-db.js:1-62`는 `GET /api/test-db`에서 사용자 조회/생성 후 매 요청마다 프로젝트와 분석을 생성한다.
- 재현: 인증 없이 엔드포인트를 두 번 호출하고 테스트 프로젝트와 분석 수가 증가하는지 확인한다.
- 권장 수정: 프로덕션 배포 대상에서 테스트·진단 함수를 제거한다. 데이터 생성 검증은 별도 테스트 DB와 로컬 스크립트에서만 수행한다.
- 회귀 테스트: 프로덕션 빌드/배포에서 `/api/test-db`, `/api/test-gemini`, `/api/test`가 404인지 확인한다.

### BLOCK-04 — 클라이언트 제공 `userId` 신뢰와 피드백 조작

- 심각도: 높음
- 영향: 임의 UUID 사용자 생성, 다른 사용자 가장, 임의 분석에 대한 피드백 생성·변경, 사용자 테이블 오염
- 증거:
  - `client/src/components/FeedbackSection.tsx:47-64`는 로컬 저장소 UUID를 `userId`로 보낸다.
  - `api/feedback.js:23-79`는 해당 값을 인증 없이 사용자 ID와 피드백 소유자로 사용한다.
  - 분석 존재만 확인하고 분석 소유자와 요청자의 관계를 확인하지 않는다.
- 재현: 임의 UUID와 알려진 `analysisId`를 POST하고, 같은 UUID로 평점을 변경한다.
- 권장 수정: 로그인 베타에서는 요청 본문의 `userId`를 제거하고 토큰의 사용자 ID를 사용한다. 피드백 대상 분석에 대한 접근 권한을 검증하고 코멘트 길이·문자열 타입을 제한한다.
- 회귀 테스트: 본문에 다른 `userId`를 넣어도 무시되거나 400, 타 사용자 분석=404, 과대 코멘트=400을 확인한다.

### BLOCK-05 — 입증되지 않은 RLS와 클라이언트 전용 관리자 권한

- 심각도: 치명적(원격 RLS가 실제로 엄격하게 구성된 경우 완화 가능)
- 영향: 사용자·자소서·피드백·AI 비용 데이터의 직접 조회 또는 변경, 관리자 역할 변조 가능성
- 증거:
  - `prisma/migrations/add_user_role.sql:16-22`의 RLS 활성화와 정책 생성문은 모두 주석이다.
  - Prisma 스키마의 `User`에는 `role`, `profile_image`, `provider`가 없지만 클라이언트는 이 컬럼을 사용한다.
  - 관리자 훅은 브라우저의 Supabase anon 클라이언트에서 테이블을 직접 조회하며 서버 관리자 API가 없다.
  - 저장소에서 `projects`, `analyses`, `feedbacks`, `token_usages`의 RLS 정책을 찾지 못했다.
- 재현: 스테이징에서 일반 사용자 토큰으로 각 테이블을 직접 SELECT/UPDATE하고 다른 사용자 행과 `users.role` 접근 여부를 확인한다.
- 권장 수정: 모든 테이블에 RLS를 활성화하고 사용자 소유 정책과 `WITH CHECK`를 명시한다. 관리자 판정은 변경 불가능한 서버 측 권한 또는 안전한 SQL 함수로 수행한다. 마이그레이션을 코드로 관리하고 Prisma 모델을 실제 스키마와 일치시킨다.
- 회귀 테스트: anon·일반 사용자·관리자별 SELECT/INSERT/UPDATE/DELETE 정책 매트릭스를 DB 통합 테스트로 고정한다.

### BETA-01 — 분석 결과가 DB에 저장되지 않아 핵심 여정이 단절됨

- 심각도: 높음
- 증거: 정상 분석 경로는 `client/src/pages/Analyze.tsx:520-527`에서 브라우저 저장소만 갱신한다. 운영 API에서 프로젝트·분석을 생성하는 코드는 없고 생성 코드는 테스트 API에만 있다.
- 영향: 분석 후 `/my`에서 영구 이력을 조회할 수 없고 브라우저 데이터 삭제·기기 변경 시 결과를 잃는다.
- 권장 수정: 인증된 분석 요청에 프로젝트/분석 PENDING 생성, 성공/실패 상태 갱신, 토큰 사용 기록을 하나의 명시적 데이터 흐름으로 구현한다.
- 회귀 테스트: 성공·타임아웃·429·파싱 실패·재시도에서 DB 상태와 비용 기록이 정확히 한 번만 남는지 확인한다.

### BETA-04 — 자소서 원문과 AI 결과의 장기 브라우저 저장

- 심각도: 중간~높음
- 증거: `client/src/utils/storage.ts:100-139`는 자소서 원문과 AI 결과를 만료시간 없이 `localStorage`와 `sessionStorage` 양쪽에 저장한다. 로그아웃 흐름은 분석 저장소를 정리하지 않는다.

- 회귀 테스트: 로그아웃, TTL 만료, 계정 삭제 후 관련 키와 화면 복원이 불가능한지 확인한다.

## 6. 수정 우선순위

### P0 — 베타 전에 반드시 완료

1. 공통 서버 인증과 모든 데이터 쿼리의 소유권 검증을 구현한다.
2. Supabase 전 테이블 RLS 정책을 코드화하고 A/B/관리자 정책 테스트를 통과시킨다.
3. 테스트·임의 Gemini API를 프로덕션에서 제거한다.
4. 분석 API에 베타 사용자 제한, 속도·비용 한도, 중복 방지와 중단 스위치를 적용한다.
5. 피드백 `userId`를 토큰 기반으로 교체하고 분석 접근 권한을 검사한다.
6. 개인정보 처리·AI 전달·보관·삭제 정책과 실제 삭제 절차를 마련한다.

### P1 — 베타 시작 전에 강력 권고

1. 분석 결과의 서버 영구 저장과 실패 상태·비용 기록을 완성한다.
2. 운영 목 폴백과 UI 전용 삭제를 제거한다.
3. 브라우저의 자소서 장기 저장을 제거하거나 TTL·로그아웃 정리를 적용한다.
4. 타입 오류를 모두 해결하고 CI 게이트를 추가한다.
5. 보안 헤더, 개인정보 제거 로그와 핵심 알림을 구성한다.

### P2 — 베타 중 개선

1. 번들 분할과 모바일 초기 로딩 성능을 개선한다.

3. 키보드, 포커스, 스크린리더와 모바일 핵심 여정 테스트를 자동화한다.
